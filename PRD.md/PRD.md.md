이 프로젝트의 목적은 사용자가 복잡하거나 번거로운 작업을 웹에서 더 간단하게 처리할 수 있도록 하는 것이다. 초기 버전에서는 꼭 필요한 핵심 기능만 우선 구현하고, 이후 실제 사용 데이터를 바탕으로 점진적으로 확장한다.

## 해결 방향
이 프로젝트는 위 문제를 해결하기 위해 아래 방향으로 설계한다.
* 핵심 작업을 짧은 단계로 줄인다.
* 직관적인 UI로 초보자도 쉽게 사용할 수 있게 한다.
* 인증, 권한, 데이터 보호를 기본 구조에 포함한다.
* 모바일과 데스크톱에서 모두 무리 없이 사용할 수 있게 한다.

## 3. 목표
## 1차 목표
1. 사용자가 서비스의 핵심 기능을 빠르게 이해하고 사용할 수 있게 한다.
2. 꼭 필요한 기능부터 먼저 안전하게 구현한다.
3. 초보 사용자도 헷갈리지 않도록 흐름을 단순하게 만든다.
4. 보안이 필요한 기능에는 인증, 권한, 데이터 보호를 적용한다.
5. 운영자도 기본적인 관리 기능을 쉽게 사용할 수 있게 한다.
## 성공 기준 요약
* 핵심 기능이 정상 작동한다.
* 보호된 기능은 인증 없이 접근할 수 없다.
* 사용자 데이터는 본인 권한 범위 내에서만 접근 가능하다.
* 모바일에서도 기본 사용이 가능하다.
* 민감정보가 프론트엔드에 노출되지 않는다.

## 4. 제외 범위 (초기 MVP에서 하지 않을 것)
초기 버전에서는 아래 기능은 우선 제외한다.
* 과도하게 복잡한 관리자 기능
* 고급 통계 대시보드
* 과한 개인화 추천
* 실시간 채팅
* 복잡한 알림 자동화
* 다중 외부 서비스 대규모 연동
* 지나치게 많은 설정 옵션
* 게임화 요소
* 불필요한 애니메이션 중심 기능
## 제외 이유
처음부터 너무 많은 기능을 넣으면 구조가 복잡해지고, 오류가 늘며, 개발 속도가 느려진다. 따라서 **핵심 기능이 안전하게 작동하는 상태**를 먼저 완성한다.

## 10. 비기능 요구사항
## 성능
* 첫 화면은 과도하게 무겁지 않아야 한다.
* 모바일 환경에서도 기본 기능이 원활해야 한다.
* 불필요한 대용량 로딩을 피한다.
## 사용성
* 초보자도 바로 이해할 수 있어야 한다.
* 버튼과 흐름은 단순해야 한다.
* 한 화면에 너무 많은 기능을 몰아넣지 않는다.
## 반응형
* 모바일 우선으로 설계한다.
* 태블릿과 데스크톱에서도 레이아웃이 깨지지 않아야 한다.
## 접근성
* 버튼, 링크, 입력창이 분명히 구분되어야 한다.
* 색 대비가 너무 약하지 않아야 한다.
* 키보드 사용도 기본적으로 가능해야 한다.
## 유지보수성
* 파일 구조가 명확해야 한다.
* 공통 컴포넌트는 재사용 가능하게 만든다.
* API와 UI 책임을 분리한다.

## 11. 보안 요구사항
이 프로젝트는 **기능만 작동하는 코드가 아니라, 보안이 적용된 상태에서 작동하는 코드**를 완료 기준으로 본다.
## 기본 보안 원칙
* 인증이 필요한 모든 API는 비즈니스 로직보다 먼저 인증 검증을 수행한다.
* 인증 여부가 불분명한 API는 보호 대상 API로 간주한다.
* 권한이 불분명하면 허용이 아니라 거부를 기본값으로 한다.
* DB 쿼리에는 반드시 검증된 user.id만 사용한다.
* req.body.userId, req.query.userId, params.userId 등 클라이언트가 보낸 사용자 식별값은 신뢰하지 않는다.
* role, plan, subscription_status, is_admin, credits 같은 민감 컬럼은 사용자 입력에서 직접 받지 않는다.
* INSERT / UPDATE 시에는 허용된 필드만 화이트리스트 방식으로 저장한다.
## 입력값 보안
* 모든 입력값은 타입, 길이, 형식 검증을 수행한다.
* SQL은 문자열 이어붙이기가 아니라 파라미터 바인딩을 사용한다.
* 예상하지 않은 필드는 거부하거나 무시한다.
## 인증 보안
* 쿠키 인증이면 httpOnly, secure, sameSite를 적용한다.
* CSRF 대응을 고려한다.
* Bearer 토큰 방식이면 서명, 만료시간, 위조 여부를 검증한다.
* 인증 실패 시 즉시 401을 반환한다.
## 데이터 보안
* 비밀번호는 평문 저장하지 않는다.
* 필요한 최소 데이터만 조회한다.
* 다른 사용자의 데이터는 권한 없이 조회/수정/삭제할 수 없어야 한다.
## 로그 / 에러 보안
* 로그와 에러 메시지에 토큰, 쿠키, 비밀번호, API 키, 민감정보를 남기지 않는다.
* 클라이언트 응답에 내부 예외, 스택 트레이스, SQL 에러를 노출하지 않는다.
## 파일 업로드 보안
* 확장자, MIME 타입, 크기를 검증한다.
* 실행 파일 업로드를 차단한다.
* 파일 접근 권한을 분리한다.
## 외부 서비스 보안
* API 키는 환경변수로만 관리한다.
* 민감 키에 NEXT_PUBLIC_ 접두사를 사용하지 않는다.
* 종량제 API는 반드시 서버에서 호출한다.
* OpenAI, Firebase, Supabase 등은 예산 한도와 알림을 설정한다.
## DB 보안
* Supabase / PostgreSQL 사용 시 모든 테이블에 RLS를 활성화한다.
* service_role 키는 서버에서만 사용한다.
* anon 키가 민감 데이터에 직접 접근하지 못하게 정책을 설정한다.

* 중요한 액션은 기록을 남길 수 있어야 한다.
* 비용이 드는 외부 API는 사용량을 모니터링할 수 있어야 한다.
* 배포 후에도 빠르게 수정할 수 있는 구조여야 한다.

## 16. 성공 기준
이 MVP는 아래 조건을 만족하면 1차 완료로 본다.
1. 사용자가 서비스 목적을 이해하고 핵심 기능을 사용할 수 있다.
2. 회원가입 / 로그인 / 로그아웃이 정상 작동한다.
3. 로그인한 사용자는 자신의 데이터만 생성/조회/수정/삭제할 수 있다.
4. 비로그인 사용자는 보호된 기능에 접근할 수 없다.
5. 기본 UI가 모바일과 데스크톱에서 모두 usable 하다.
6. 주요 API에 입력 검증과 보안 규칙이 적용되어 있다.
7. 민감한 키와 인증정보가 프론트엔드에 노출되지 않는다.
8. 관리자 기능이 일반 사용자 기능과 분리되어 있다.

## 19. 코드 작성 후 필수 자체 점검
* 인증 실패 시 즉시 401을 반환했는가?
* 인증 검증이 비즈니스 로직보다 먼저 실행되는가?
* DB 쿼리에 검증된 user.id만 사용했는가?
* 클라이언트가 보낸 userId를 신뢰하지 않았는가?
* 민감 컬럼이 사용자 입력에서 오지 않는가?
* INSERT / UPDATE 시 화이트리스트 방식만 사용했는가?
* 입력값의 타입, 길이, 형식 검증을 했는가?
* SQL 파라미터 바인딩을 적용했는가?
* SELECT / UPDATE / DELETE 시 소유권 검사를 했는가?
* 쿠키 방식이면 httpOnly, secure, sameSite를 설정했는가?
* 쿠키 방식이면 CSRF를 고려했는가?
* Bearer 방식이면 토큰 서명/만료 검증을 했는가?
* 비밀번호를 안전한 해시로 저장하는가?
* 로그와 에러 응답에 민감정보가 남지 않는가?
* 관리자 권한 검사가 일반 사용자와 분리되어 있는가?
* 파일 업로드 시 확장자, MIME, 크기 검증을 했는가?
* RLS가 필요한 테이블에 적용되어 있는가?
* API 키가 프론트엔드에 노출되지 않는가?
* 종량제 서비스 예산 한도와 알림이 설정되어 있는가?
