# AGENTS.md

이 프로젝트에서는 기능 구현만으로 완료로 보지 않는다.
**인증, 권한, 입력 검증, 데이터 보호가 적용된 상태에서 안전하게 작동하는 코드만 완료로 간주한다.**

---

## 목적

이 문서는 이 프로젝트에서 코드를 작성하거나 수정할 때 반드시 따라야 할 보안 기준을 정의한다.
모든 기능 개발, API 작성, DB 접근, 외부 서비스 연동은 아래 원칙을 기본값으로 따른다.

---

## 핵심 보안 원칙

- 인증이 필요한 모든 API는 **비즈니스 로직보다 먼저 인증 검증**을 수행한다.
- 인증 여부가 불분명한 API는 **보호 대상 API로 간주**하고 인증 검증을 먼저 적용한다.
- 권한이 명확하지 않으면 **허용하지 말고 거부를 기본값**으로 한다.
- DB 쿼리에는 오직 **검증된 `user.id`** 만 사용한다.
- 클라이언트가 보낸 사용자 식별값은 신뢰하지 않는다.
- 민감한 값은 사용자 입력이 아니라 **서버 로직으로만 통제**한다.
- 저장 시에는 항상 **허용된 필드만 화이트리스트 방식으로 반영**한다.
- 프론트엔드는 표시와 입력만 담당하고, **민감한 판단과 처리는 반드시 서버에서 수행**한다.

---

## 인증 및 권한 규칙

### 1) 쿠키 세션 방식
- 인증이 필요한 API는 비즈니스 로직보다 먼저 아래 순서로 검증한다.
  1. `req.cookies.get("session")`
  2. `verifySession(session)`
  3. 실패 시 즉시 `401 Unauthorized` 반환

- 인증 실패 후에는 어떤 비즈니스 로직도 실행하지 않는다.
- 세션 쿠키에는 반드시 아래 속성을 적용한다.
  - `httpOnly`
  - `secure`
  - `sameSite`
- 쿠키 기반 인증을 사용할 때는 **CSRF 대응**을 반드시 고려한다.

### 2) Bearer 토큰 방식
- 인증이 필요한 API는 비즈니스 로직보다 먼저 아래 순서로 검증한다.
  1. `Authorization: Bearer <token>` 확인
  2. `verifyAccessToken(token)` 검증
  3. 실패 시 즉시 `401 Unauthorized` 반환

- Bearer 토큰은 반드시 아래를 검증한다.
  - 서명 유효성
  - 만료시간
  - 위조 여부
  - 필요한 경우 issuer / audience

### 3) 권한 검사
- 인증과 권한은 다르다. 로그인되어 있어도 모든 기능을 사용할 수 있는 것은 아니다.
- 관리자 기능은 일반 사용자 기능과 분리해서 검사한다.
- 일반 사용자는 다른 사용자의 데이터를 조회, 수정, 삭제할 수 없어야 한다.
- `SELECT`, `UPDATE`, `DELETE` 수행 시 대상 데이터가 **검증된 `user.id` 소유인지 반드시 확인**한다.

---

## 사용자 식별값 처리 규칙

- DB 쿼리에는 반드시 **검증된 인증 결과의 `user.id`** 만 사용한다.
- 아래 값들은 사용자 식별 근거로 사용하지 않는다.
  - `req.body.userId`
  - `req.query.userId`
  - `params.userId`
  - 프론트엔드가 전달한 임의의 사용자 ID
- “누구의 데이터인지”는 항상 서버가 인증 결과로 판단한다.

쉽게 말하면,
사용자가 종이에 자기 이름을 써서 가져왔다고 해서 본인 확인이 끝난 게 아니다.
**신분증을 서버가 직접 확인한 뒤, 그 확인된 정보만 써야 한다.**

---

## 민감 컬럼 처리 규칙

아래와 같은 민감 컬럼은 절대 사용자 입력에서 직접 받지 않는다.

- `role`
- `plan`
- `subscription_status`
- `is_admin`
- `credits`
- 결제 상태
- 관리자 여부
- 내부 권한 값
- 사용량 제한 관련 값

이 값들은 반드시 **서버 로직, 관리자 승인, 결제 검증 결과, 내부 정책**으로만 결정한다.

---

## 입력값 검증 규칙

모든 입력값은 신뢰하지 않는다.
항상 아래를 검증한다.

- 타입 검증
- 필수값 여부
- 길이 제한
- 형식 검증
- 허용 범위 검증
- 예상하지 않은 필드 차단

예를 들면:
- 이메일 → 이메일 형식 검사
- 닉네임 → 최소/최대 길이 검사
- 숫자 → 숫자인지와 허용 범위 검사
- 날짜 → 올바른 날짜 형식인지 검사

또한 아래 원칙을 따른다.

- `req.body` 전체를 그대로 저장하지 않는다.
- `INSERT` / `UPDATE` 시 허용된 필드만 골라서 저장한다.
- 알 수 없는 필드는 무시하거나 거부한다.

쉽게 말하면,
택배 상자를 통째로 창고에 넣지 말고,
**안에 든 물건을 하나씩 검사한 뒤 허용된 것만 넣는 방식**으로 처리해야 한다.

## DB 보안 규칙

### 공통 규칙
- SQL은 문자열 이어붙이기로 만들지 않는다.
- 반드시 **파라미터 바인딩**을 사용한다.
- 조회, 수정, 삭제 시 항상 사용자 소유권과 권한을 함께 확인한다.
- 민감 데이터는 필요한 경우에만 최소 범위로 조회한다.

### Supabase / PostgreSQL 규칙
- 모든 테이블에 **RLS(Row Level Security)** 를 활성화한다.
- `service_role` 키는 **서버(백엔드)에서만 사용**한다.
- `anon` 키로는 민감 테이블에 접근할 수 없도록 정책을 설정한다.
- 다른 사용자의 데이터를 `SELECT`, `UPDATE`, `DELETE` 할 수 없도록 정책을 검증한다.
- RLS가 켜져 있어도 서버 코드에서 소유권 검사를 생략하지 않는다.

쉽게 말하면,
RLS는 건물의 공용 출입문 잠금장치이고,
서버의 소유권 검사는 각 방 문 앞의 추가 잠금장치다.
**둘 다 있어야 더 안전하다.**

---

## 비밀번호 및 인증정보 보안

- 비밀번호는 평문으로 저장하지 않는다.
- 안전한 해시 알고리즘으로 저장한다.
- 토큰, 쿠키, 비밀번호, API 키, 인증정보는 로그에 남기지 않는다.
- 에러 메시지에도 민감정보를 포함하지 않는다.
- 클라이언트 응답에 내부 인증 구조를 자세히 노출하지 않는다.

---

## 에러 및 로그 처리 규칙

- API 에러 응답에 아래 정보를 노출하지 않는다.
  - 스택 트레이스
  - SQL 원문
  - 내부 예외 메시지
  - 토큰 상세 상태
  - 세션 값
  - 비밀번호
  - API 키
- 클라이언트에는 일반화된 메시지로 응답한다.
- 상세 원인은 서버 내부 로그에서만 확인한다.
- 로그에는 민감정보를 마스킹하거나 기록하지 않는다.

쉽게 말하면,
문이 잠겨 있는 것뿐 아니라
**왜 잠겼는지 내부 구조를 바깥 사람에게 자세히 설명하지 않는 것**도 보안이다.

---

## Rate Limit 및 남용 방지

다음 기능에는 rate limit를 적용한다.

- 로그인
- 회원가입
- 비밀번호 재설정
- 인증 코드 요청
- 주요 비용 발생 API
- 외부 종량제 API 호출
- 파일 업로드
- 관리자 기능

가능하면 아래도 함께 고려한다.

- IP 기반 제한
- 계정 기반 제한
- 짧은 시간 반복 요청 차단
- 비정상 패턴 탐지

---

## 파일 업로드 보안 규칙

파일 업로드 기능이 있으면 반드시 아래를 검증한다.

- 확장자
- MIME 타입
- 파일 크기
- 업로드 개수 제한
- 저장 위치
- 실행 가능한 파일 차단 여부

추가 원칙:
- 파일명은 그대로 신뢰하지 않는다.
- 필요하면 서버에서 안전한 이름으로 변경한다.
- 공개 업로드와 비공개 업로드를 구분한다.
- 업로드 후 접근 권한을 따로 검사한다.

---

## 외부 서비스 보안 규칙

- OpenAI, Firebase, Supabase 등 종량제 서비스는 콘솔에서 **예산 한도와 알림**을 반드시 설정한다.
- API 키는 환경변수로만 관리하고 코드에 직접 작성하지 않는다.
- 민감한 키에는 `NEXT_PUBLIC_` 접두사를 절대 사용하지 않는다.
- 모든 민감한 외부 API 호출은 **백엔드(서버)에서만 수행**한다.
- 결제, 관리자 기능, 사용자 데이터 조회/수정, 종량제 API 호출은 **반드시 서버를 경유**해야 한다.
- 프론트엔드 코드에서 비밀키, 관리자 키, 서비스 키가 노출되지 않도록 한다.

---

## HTTPS 및 전송 보안

- 운영 환경에서는 HTTPS를 기본으로 사용한다.
- 민감한 인증정보는 암호화되지 않은 연결로 전송하지 않는다.
- 보안 쿠키는 HTTPS 환경에서만 안전하게 사용되도록 설정한다.

---

## Next.js / 프론트엔드 보안 규칙

- 민감한 로직은 클라이언트 컴포넌트에 넣지 않는다.
- 권한 검사만 프론트엔드에서 하고 서버 검사를 생략하면 안 된다.
- 프론트엔드의 조건문은 UX용일 뿐, 보안 경계가 아니다.
- 최종 권한 확인은 항상 서버에서 한다.
- API 키, 관리자 키, 서비스 키, 내부 URL 규칙 등 민감정보가 번들에 포함되지 않도록 한다.

쉽게 말하면,
프론트엔드의 숨김 버튼은 커튼일 뿐이고,
실제 자물쇠는 서버에 있어야 한다.

## 코드 작성 후 필수 자체 점검

코드를 제출하기 전에 반드시 아래를 스스로 확인한다.

- [ ] 인증 실패 시 즉시 `401`을 반환했는가?
- [ ] 인증 검증이 비즈니스 로직보다 먼저 실행되는가?
- [ ] DB 쿼리에 검증된 `user.id`만 사용했는가?
- [ ] 클라이언트가 보낸 `userId`를 신뢰하지 않았는가?
- [ ] 민감 컬럼이 사용자 입력에서 오지 않는가?
- [ ] `INSERT` / `UPDATE` 시 화이트리스트 방식만 사용했는가?
- [ ] 입력값의 타입, 길이, 형식 검증을 했는가?
- [ ] SQL 파라미터 바인딩을 적용했는가?
- [ ] `SELECT` / `UPDATE` / `DELETE` 시 소유권 검사를 했는가?
- [ ] 쿠키 방식이면 `httpOnly`, `secure`, `sameSite`를 설정했는가?
- [ ] 쿠키 방식이면 CSRF를 고려했는가?
- [ ] Bearer 방식이면 토큰 서명/만료 검증을 했는가?
- [ ] 비밀번호를 안전한 해시로 저장하는가?
- [ ] 로그와 에러 응답에 민감정보가 남지 않는가?
- [ ] 관리자 기능과 일반 사용자 기능의 권한 검사가 분리되어 있는가?
- [ ] 파일 업로드 시 확장자, MIME, 크기 검증을 했는가?
- [ ] 모든 테이블에 RLS가 활성화되어 있는가?
- [ ] `service_role` 키가 서버에서만 사용되는가?
- [ ] API 키가 프론트엔드 코드에 노출되지 않는가?
- [ ] 종량제 서비스 예산 한도와 알림이 설정되어 있는가?

## 최종 원칙

보안은 기능 개발이 끝난 뒤 나중에 붙이는 옵션이 아니다.
**처음부터 코드 구조 안에 들어가 있어야 하는 기본 조건**이다.

이 프로젝트에서 좋은 코드는
단순히 “작동하는 코드”가 아니라,
**올바른 사용자만, 올바른 방식으로, 허용된 데이터에만 접근하도록 통제된 코드**이다.
